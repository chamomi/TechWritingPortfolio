SSL: How It Works

What Is SSL?

Secure Sockets Layer (SSL) is a network security protocol. It creates encrypted connection sessions to ensure privacy, authentication, and integrity for Internet communication. SSL is also referred to as SSL/TLS.

SSL serves to:

  • Ensure privacy of the data transmitted through the public Internet.
    SSL encrypts the transmitted data so that only the receiver can read it. SSL minimizes the risk of interception of the sensitive data.
  • Authenticate the web-server.
    The SSL certificate confirms the trust-worthiness of the web server that the user is trying to connect to.
  • Provide data integrity.
    SSL protects the data from modifications during the transmission.

SSL protects sensitive user data such as bank account information, personal data, login credentials and many more. That is why it is an important part of the modern Internet.

What Is SSL Certificate?

SSL certificate is a document installed on the server. The certificate confirms the website’s identity and contains the SSL public key to establish encrypted connections. SSL protection works only with the servers that have an SSL certificate. The certificate can be self-signed or issued by one of the trusted certificate authorities. Self-signed certificates can be used for encryption, however, the browser may raise a warning for a suspicious certificate source.

SSL connection starts with the SSL handshake. During the handshake, the client decides if the server is trust-worthy based on its SSL certificate.

How SSL Handshake Is Performed?

SSL handshake is the process that establishes a secure, encrypted connection between the client and the server. SSL session is protected by a session key that is the same for the client and the server. SSL uses public key encryption to generate the session key and establish the secure channel. Public key encryption uses a pair of public and private keys to encrypt and decrypt data. The data encrypted with one key from the pair can only be decrypted with another key from the pair.

The SSL handshake consists of the following stages:

  1. The client initiates the SSL handshake.
  2. The server sends its SSL certificate to the client. The certificate includes the server’s public key.
  3. The client verifies the server’s SSL certificate. If the certificate is not issued by one of the trusted certificate authorities, the user will receive a warning. The user can accept a certificate from a not trusted source at his own risk.
  4. The client generates a random session key and encrypts it with the server’s public key.
  5. The client sends the encrypted session key to the server.
  6. The server decrypts the session key with its private key. If the SSL certificate of the server is forged, the server will not be able to decrypt the session key and the handshake will fail, keeping user data safe.
  7. SSL handshake finishes with an established encrypted connection protected by the session key known both to the client and the server.

After the successful completion of the SSL handshake, the client and the server exchange user data encrypted with the session key. Each new session requires a new session key. So all new connection requests trigger the SSL handshake again.
